# RAG 在线检索流水线技术方案（Retrieval Pipeline）

## 1. 概述

在线检索流水线负责接收用户查询，经过 Query 预处理、多路召回、重排序、上下文构造、Prompt 组装后调用 LLM 生成最终回答。

```
Query → 预处理 → 多路召回 → 重排 → 上下文构造 → Prompt → LLM → 后处理 → 用户
```

---

## 2. Query 预处理模块（Query Preprocessing）

提升用户原始查询的质量，决定后续召回效果。

| 处理环节 | 功能 | 技术方案 |
|---------|------|---------|
| 清洗 | 去除无意义符号、纠错 | 正则 + 拼写检查 |
| Query Rewrite | 扩展同义词、补全语义 | LLM 改写 / 规则映射 |
| 意图识别 | 区分事实查询、总结类、对比类 | 分类模型 / LLM 判断 |
| 多查询生成 | 一题多解，生成多个子查询 | LLM 生成 + 验证 |

**设计要点**：
- 改写结果可配置是否展示给用户（透明化）
- 缓存高频 query 的预处理结果，降低延迟

---

## 3. 多路召回模块（Multi-Channel Retrieval）

从多个索引通道召回候选文档，扩大覆盖范围。

| 召回通道 | 召回方式 | 作用 |
|---------|---------|------|
| Dense Retrieval | 向量相似度 TopK | 语义相似召回 |
| Sparse Retrieval | BM25 / TF-IDF TopK | 关键词精准召回 |
| FAQ 召回 | 精确匹配 / 语义匹配 | 标准问答直达 |
| 缓存召回 | 热点 query 结果缓存 | 降低延迟 |
| 图谱召回（可选） | 实体关系扩展 | 知识图谱关联 |

**融合策略**：
- **RRF（Reciprocal Rank Fusion）**：按排名位置加权融合，无需校准分数
- **加权融合**：为不同通道分配权重，线性加权求和
- **置信度过滤**：单路分数过低的结果直接丢弃

---

## 4. 重排模块（Rerank Module）

对融合后的候选集进行精排，提升 TopK 结果的相关性。

- **Cross-Encoder**：query + doc 拼接输入，输出相关性分数
  - 模型：bge-reranker、Cohere Rerank
  - 特点：精度高、计算量大，仅对 TopK（如 100→20）重排
- **粗排+精排**：先轻量模型粗筛，再 Cross-Encoder 精排

**设计要点**：
- 重排后截取 TopN（如 Top5/Top10）作为最终上下文
- 分数阈值过滤，低于阈值的文档丢弃，避免污染上下文

---

## 5. 上下文构造模块（Context Construction）

将重排后的文档组织为 LLM 可消费的上下文格式。

- **去重**：相同/相似内容合并，避免重复信息
- **排序**：按相关性分数降序排列
- **截断**：按 token 预算截断，保留最相关的部分
- **编号**：为每个 chunk 分配序号，便于 LLM 生成引用标记
- **元数据注入**：在 chunk 前附加来源信息（文件名、章节）

**上下文模板示例**：
```
[1] 来源：产品手册_v2.3.pdf | 章节：安装指南
内容：首先下载安装包，然后执行 setup.exe ...

[2] 来源：FAQ数据库 | 分类：常见问题
内容：如果安装失败，请检查系统版本是否满足要求 ...
```

---

## 6. Prompt 构建模块（Prompt Builder）

将系统指令、上下文、用户查询组装为最终 Prompt。

- **系统指令（System Prompt）**：定义模型角色、回答风格、约束条件
- **上下文（Context）**：检索得到的参考文档
- **用户查询（User Query）**：原始问题或改写后的问题

**设计要点**：
- 支持 Prompt 模板化管理，按场景切换（问答 / 总结 / 对比）
- 控制总 token 数，预留生成预算（Context + Query + 回答 < 模型窗口）
- 注入防幻觉指令：要求基于提供的上下文回答，不确定时明确告知

---

## 7. LLM 生成模块（Generation Module）

调用大语言模型生成最终回答。

| 部署方式 | 特点 | 适用场景 |
|---------|------|---------|
| 云端 API | GPT-4 / Claude / 文心 / 通义 | 快速上线、高质量要求 |
| 私有化部署 | Llama / ChatGLM / Qwen | 数据安全、离线环境 |
| 本地轻量 | Ollama + 小参数模型 | 开发测试、边缘场景 |

**设计要点**：
- 封装 `LlmClient` 接口，统一 chat/completion/stream 调用方式
- 支持流式返回（SSE），提升用户体验
- 配置温度、最大token、超时、重试策略
- 接入模型路由：按问题类型/成本/负载选择不同模型

---

## 8. 后处理模块（Post-Processing）

对 LLM 输出进行加工，提升可用性。

| 处理环节 | 功能 |
|---------|------|
| 引用补充 | 将回答中的 `[1]` `[2]` 替换为实际来源链接 |
| 格式化 | Markdown 美化、表格渲染、代码高亮 |
| 敏感校验 | 过滤敏感内容、合规检查 |
| 置信度标记 | 对不确定的内容标注提示 |
| 日志记录 | 记录 query → recall → answer 全链路，用于效果分析 |

---

## 9. 在线数据流完整示意

```
用户输入 Query
    ↓
Query 预处理（清洗 / 改写 / 意图识别）
    ↓
并行多路召回：
    ├── Dense Retrieval（向量 TopK）
    ├── Sparse Retrieval（BM25 TopK）
    └── 其他召回（FAQ / 缓存）
    ↓
召回结果融合（RRF / 加权）
    ↓
重排模块（Reranker TopN）
    ↓
上下文构造（去重 / 截断 / 编号）
    ↓
Prompt 组装（指令 + 上下文 + Query）
    ↓
LLM 生成（流式输出）
    ↓
后处理（引用补充 / 格式化 / 校验）
    ↓
返回用户
```

---

## 10. 接口规划

| 模块 | 核心接口 | 职责 |
|-----|---------|------|
| Query 预处理 | `QueryPreprocessor.process()` | 清洗 / 改写 / 意图识别 |
| 向量检索 | `VectorStore.search()` | 向量相似度检索 |
| 倒排检索 | `BM25Store.search()` | BM25 关键词检索 |
| 检索编排 | `RetrievalService.retrieve()` | 多路召回 + 融合 + 重排 |
| 重排 | `Reranker.rerank()` | 精排候选文档 |
| 生成 | `LlmClient.chat()` | 调用 LLM 生成回答 |
| 问答门面 | `RagService.answer()` | 对外统一问答接口 |

---

## 11. 非功能性设计

| 指标 | 目标值 | 说明 |
|-----|-------|------|
| 端到端延迟（P99） | < 3s | 不含 LLM 首 token 时间 |
| 检索延迟（P99） | < 500ms | 召回 + 重排 |
| 吞吐 | > 100 QPS | 在线查询 |

**可靠性**：
- 向量服务降级：主索引故障时切换备用索引或走稀疏通道兜底
- LLM 调用熔断：超时/报错时返回友好提示或 fallback 模型

**可观测性**：
- 全链路追踪：从 query 输入到 answer 输出，记录每步耗时和中间结果
- 效果评估：定期抽样人工评估，计算回答准确率、召回率

---

## 12. 演进路线

| 阶段 | 目标 | 关键能力 |
|-----|------|---------|
| MVP | 单路向量召回 + 云端 LLM | 跑通核心链路 |
| V1 | 混合召回 + 重排 | 提升召回质量 |
| V2 | Query 改写 + 意图识别 + 缓存 | 降低延迟、提升体验 |
| V3 | 本地 LLM + 流式输出 | 私有化、降低成本 |
| V4 | 知识图谱 + Agent 能力 | 复杂推理、多跳问答 |
